## 1. 项目介绍话术

### 30 秒电梯演讲（一面开场）

> "我独立开发了一个叫 Clarity 的 AI Agent 运行时，用纯 Rust 写的。它是一个本地优先的 AI 编程助手，支持 OpenAI、Claude、DeepSeek 等云端模型，也可以用 Candle 框架直接跑本地 GGUF 模型——零 Python 依赖。整个项目有 22 个 crate，大概 15 万行代码，1554 个测试全部通过，零 Clippy 警告。"

### 2 分钟详细版（二面深挖）

> "Clarity 解决的问题是：现有的 AI 编程工具（Copilot/Cursor/Claude Code）要么是云端的（数据安全风险），要么依赖 Python/Node.js 运行时（部署复杂）。Clarity 是一个纯 Rust 的单二进制文件，`cargo install` 就能用。
>
> 架构上我用了 Contract-First 原则，最底层的 clarity-contract crate 零内部依赖，定义了所有共享 trait，上面 20 多个 crate 都基于这些 trait 工作。Agent 引擎支持 ReAct 和 Plan 两种模式——ReAct 适合探索性任务（边想边做），Plan 适合结构性任务（先规划后执行）。工具调用有四层审批链：Interactive（每次确认）、Smart（首次确认后自动放行）、Plan（审批整个计划）、Yolo（全自动）。
>
> 存储方面，我用 SQLite + FTS5 做全文搜索，BM25 做关键词检索，向量余弦相似度做语义搜索，三者融合用 RRF 算法。没有引入任何外部向量数据库。记忆系统有四层：今天/本周/长期/事实，有点像人脑的分层记忆模型。MCP 协议我实现了全部四种传输：stdio、SSE、HTTP、WebSocket。
>
> 目前 6 个前端共享同一个 Agent 内核：egui 桌面 GUI、ratatui 终端 TUI、Axum Web 网关、系统托盘、headless CLI、移动端 FFI。1554 个测试全部通过，生产代码零 unwrap/expect/panic。"

## 2. 高频面试问题与参考答案

### Q1: 为什么选择 Contract-First 架构？（⭐⭐⭐⭐⭐ 架构设计）

**考察维度**：架构思维、依赖管理、模块化设计

**推荐回答**：
"Contract-First 的核心思想来自依赖反转原则。clarity-contract 定义了所有共享 trait——`LlmProvider`、`Tool`、`Message`、`StreamDelta` 等——但它本身零内部依赖，不对任何具体的存储引擎、网络库、ML 框架做假设。

这样做的三个好处：
1. **编译时验证**：所有 crate 在编译时就确定了接口兼容性，不需要运行时反射或动态检查
2. **可替换性**：我可以独立替换 LLM provider（OpenAI ↔ Anthropic ↔ DeepSeek）、存储引擎、MCP 传输层，只要实现了对应的 trait
3. **避免循环依赖**：contract 在最底层，所有 crate 依赖它，但它不依赖任何人——编译器天然保证不会有循环引用

在 22 个 crate 的情况下，如果没有 contract，很容易出现两个 crate 互相依赖的循环。Contract-First 强制了单向依赖。"

### Q2: ReAct vs Plan 模式的区别和选择？（⭐⭐⭐⭐⭐ Agent 设计）

**考察维度**：Agent 架构、算法理解、设计权衡

**推荐回答**：
"ReAct 模式是 Reasoning + Acting 的交替循环：模型先输出思考，然后调用工具，观察工具结果，再继续思考。它适合**探索性任务**——你事先不知道需要多少步，每一步依赖前一步的结果。

Plan 模式是先让模型生成一个完整的执行计划，然后按步骤执行，每步完成后验证结果。它适合**结构性任务**——目标明确，步骤可以预分解。

在 Clarity 中，Plan 模式有一个状态机追踪每一步的状态（pending/running/success/failed），如果某步失败，可以触发重新规划。

选择策略：如果用户说'帮我修复这个 bug'，用 ReAct；如果用户说'帮我给这个项目添加 CI/CD'，用 Plan。复杂度判断用启发式规则：任务描述中有关键词（步骤/流程/搭建）或者预期步骤 > 3 时，倾向 Plan。"

### Q3: 四层审批链的设计原理？（⭐⭐⭐⭐ 安全设计）

**考察维度**：安全思维、用户体验、信任模型

**推荐回答**：
"四层审批链对应四种信任级别：

1. **Interactive**：每次工具调用都弹窗确认。适合不信任的代码/环境，或者需要人工判断的关键操作（如 rm -rf）
2. **Smart**：同类操作首次确认后自动放行。比如第一次读文件弹窗，用户确认后，后续同目录的文件读取自动放行。这是日常开发的默认模式
3. **Plan**：审批整个执行计划，计划内的步骤自动执行。适合明确的、可预期后果的多步任务
4. **Yolo**：完全自动执行。适合完全信任的环境或低风险操作

设计考量：
- 每次工具调用都检查审批状态，不是启动时一次性决定
- Smart 模式的'同类操作'定义基于工具名 + 参数模式匹配
- 有逃逸机制：即使 Plan 模式，关键操作（如 git push）也可以强制弹窗
- 审批历史可审计：所有决策记录在 OpLog 中"

### Q4: 为什么不用向量数据库？（⭐⭐⭐⭐ 存储设计）

**考察维度**：技术选型、本地优先思维、工程权衡

**推荐回答**：
"我选择了 SQLite + 自定义 cosine_similarity UDF + BM25 + FTS5 的方案，没有引入 Qdrant、Milvus 或 Chroma。原因：

1. **本地优先**：Clarity 是本地应用，不依赖任何外部服务。引入向量数据库意味着用户需要安装和运维另一个服务
2. **规模考量**：个人使用的向量规模通常不超过百万级，SQLite + 自定义 UDF 完全够用。分布式向量数据库是为了十亿级向量设计的
3. **部署简化**：一个二进制文件解决所有问题，不需要 Docker Compose
4. **混合搜索**：BM25 处理关键词匹配（精确、可解释），向量搜索处理语义相似（模糊、全局），RRF 融合两者的排名。实际使用中，BM25 往往比纯向量搜索更准确——特别是搜索代码时

如果未来规模增长到需要专用向量数据库，架构上也容易迁移：只需实现一个新的 StorageBackend trait。"

### Q5: SPMC 事件总线 vs MPSC？（⭐⭐⭐ 通信架构）

**考察维度**：并发编程、设计模式

**推荐回答**：
"SPMC（Single Producer Multiple Consumer）的选择是因为 Clarity 有多个前端需要同时监听同一个 Agent 的事件流。

Agent 内核是单一生产者——它产生对话内容、工具调用结果、状态变更。前端是多个消费者——TUI 显示对话、桌面 GUI 更新状态栏、Web Gateway 推送给浏览器客户端。

如果使用 MPSC（Multiple Producer Single Consumer），每个消息只能被一个消费者收到，就需要额外的分发逻辑。SPMC 广播让每个前端独立决定如何消费消息——TUI 只关心对话内容，系统托盘只关心错误/告警。"

### Q6: 零 unwrap 策略的工程价值？（⭐⭐⭐ 工程质量）

**考察维度**：代码质量、团队协作、错误处理

**推荐回答**：
"零 unwrap/expect/panic 策略是用 Clippy 的 deny 规则强制执行的。三个价值：

1. **生产可靠性**：每个错误路径都被显式处理。不会有'我以为这里永远不会出错'的假设——网络会断、文件会被删、API 会限流
2. **代码审查**：reviewer 看到 `let x = fallible()?;` 就知道这里有错误处理。如果看到 unwrap，需要问'为什么这里不会出错？'
3. **新人友好**：新人加入项目时，看到 ? 就知道这段代码可能失败，看到 unwrap 可能误以为永远不会失败

唯一的例外是 unsafe 块内——但必须附带 `// SAFETY:` 注释解释为什么安全。测试代码中允许 unwrap——因为测试就是要 panic 来暴露问题。"

### Q7: Candle vs Ollama 的架构差异？（⭐⭐⭐⭐ 本地推理）

**考察维度**：技术选型、本地推理、Rust 生态

**推荐回答**：
"Ollama 是一个独立守护进程，用 Go 编写，内部封装了 llama.cpp。使用它需要：
- 安装 Ollama 二进制
- 启动 Ollama 服务
- 通过 HTTP API 调用

Candle 是 HuggingFace 的 Rust ML 框架，可以直接嵌入 Rust 应用。使用它：
- 零外部依赖，编译进同一个二进制
- 直接加载 GGUF 文件
- 不需要 HTTP 调用，函数调用即可

选择 Candle 是为了保持'单二进制'哲学。但 Candle 的生态不如 Ollama/llama.cpp 成熟——支持的模型架构少，推理性能优化不够。所以我同时支持了 Ollama HTTP 和 llama.cpp server 作为补充。"

### Q8: 22 个 crate 如何管理依赖？（⭐⭐⭐ 工程管理）

**考察维度**：工程能力、模块化设计

**推荐回答**：
"关键是严格的依赖方向控制和 Cargo workspace 管理。

依赖方向是单向的：contract → wire/memory/mcp/llm → core → subagents/telemetry → frontends。clarity-core 绝对不能依赖任何前端 crate，前端 crate 之间不能互相 import。

Cargo workspace 管理技巧：
- `[workspace.dependencies]` 统一版本管理，避免不同 crate 使用不同版本的同一个依赖
- feature flag 按能力划分（local-llm、hermes、tray），不是按 crate 划分
- CI 中 `cargo check --workspace` 确保所有 crate 编译通过
- 新增 crate 必须在 CI 中注册（fmt/clippy/test/audit 全覆盖）

如果一个前端 crate 的代码超过 5000 行，考虑提取子模块；但如果子模块有超过 5 个 `crate::` 引用，可能不该提取——RF-7 的考量。"

### Q9: 四层记忆压缩的具体策略？（⭐⭐⭐⭐ 记忆系统）

**考察维度**：记忆架构、信息压缩

**推荐回答**：
"四层结构：

1. **Today**：最近 20 个 Turn 的完整对话。不做任何压缩，保证最新的上下文完整
2. **Week**：对已完成会话生成结构化摘要——包含意图、关键决策、使用的工具、结果、未解决的问题。通过 LLM 生成
3. **Long-term**：从多个摘要中提取跨会话模式。例如'用户偏好用 Rust 实现系统工具'、'项目 X 使用 parking_lot'
4. **Facts**：最底层的原子化事实。每个事实有独立的来源、置信度、时间戳。事实之间通过关系图连接

检索时从上到下：先在 Today 找，再到 Week → Long-term → Facts。每层有独立的过期策略和压缩触发条件。"

### Q10: 如何保证 Agent 不陷入循环？（⭐⭐⭐ 可靠性）

**考察维度**：可靠系统、边缘情况

**推荐回答**：
"三重保护：
1. **硬性步数限制**：单个 Turn 最多 N 步（可配置），超过自动终止
2. **语义重复检测**：比较连续步骤的文本相似度，如果连续 3 步相似度 > 0.9，判定为循环
3. **Token 预算**：单轮总 Token 消耗有上限，达到后强制总结并终止

实际经验：硬性步数限制捕获了 90% 的循环情况，语义重复检测捕获了剩余的边缘情况。"

## 3. 技术深挖问题

### 深挖1：现场设计一个 MCP 传输层（要求画图）

**提示**："假设你要给 Clarity 添加对 Unix Domain Socket 传输的支持，你会怎么设计？"

**推荐回答要点**：
1. 实现 `Transport` trait（`connect/listen/send/receive`）
2. 处理连接生命周期（创建 → 认证 → 消息循环 → 断开）
3. 错误恢复策略（重连、超时）
4. 与现有四种传输的互操作性

### 深挖2：如何优化 Agent 的推理延迟？

**推荐回答要点**：
1. Provider 层面的自适应路由——将简单任务路由到更便宜/更快的模型
2. 工具调用批处理——将可并行的工具调用批量发送
3. 流式响应处理——不等完整响应就开始处理（streaming parse）
4. Prompt 缓存——Anthropic/OpenAI 的缓存机制可节省 90% 输入 token
