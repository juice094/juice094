## 1. 项目介绍话术

### 30 秒版

> "我用纯 Rust 重新实现了 Syncthing 的 BEP 协议——这是一个 P2P 文件同步系统。完整实现了 TLS 1.3、NAT 穿透、四种设备发现机制、块级文件同步和冲突解决。392 个测试全部通过，一个约 13MB 的静态二进制文件，零外部依赖。"

### 2 分钟版

> "Syncthing-rust 是我对 Go Syncthing 的完整重新实现，与 Go 版本线级兼容。实现了完整的 BEP 协议栈：Protobuf 编解码、TLS 1.3 + Ed25519 设备身份认证、TCP/WebSocket/DERP 多种传输层。四种发现机制——LAN UDP 广播、Global HTTPS mTLS、STUN、UPnP——加上 Relay 中继支持。
>
> 同步引擎用 SHA-256 做块级扫描，rayon 并行哈希，sled 做元数据存储，LZ4 压缩传输。冲突解决用 similar crate 做三路文本合并。自适应并发控制根据 RTT 动态调整并发请求数。已经稳定运行在我的双节点部署上好几个月。"

## 2. 高频面试问题

### Q1: 为什么要重新实现 Go Syncthing？（⭐⭐⭐⭐⭐）

**推荐回答**：
"三个原因：
1. **技术学习**：重新实现一个已知协议的完整系统是理解 P2P/分布式系统的最佳方式。深入理解了 BEP 协议设计、NAT 穿透、加密通信、并发控制
2. **工程需求**：需要一个可以嵌入 Rust 项目的同步方案。Go 服务需要独立部署和通信开销，Rust 库可以直接 link
3. **证明能力**：从协议规范到可运行系统，证明有独立交付复杂系统的能力

这不是'重复造轮子'——我是在验证自己能否将一个复杂的分布式协议从零实现到生产可用。"

### Q2: BEP 协议的核心消息流？（⭐⭐⭐⭐⭐）

**推荐回答**：
"五种核心消息：
1. **Hello**：连接建立后第一个消息，交换设备 ID。对端验证设备 ID 与 TLS 证书是否一致
2. **ClusterConfig**：交换文件夹配置。告诉对方有哪些文件夹，各自的 ID、标签、权限
3. **Index**：交换文件列表。本地文件夹的文件，每个文件的 SHA-256 块哈希列表
4. **Request**：请求缺失的块。需要文件夹 X 中文件 Y 的第 3-7 个块
5. **Response**：返回请求的块数据。LZ4 压缩后传输

消息流：Hello（认证）→ ClusterConfig（配置协商）→ Index（文件列表对比）→ Request/Response（块同步）

与 BitTorrent 的区别：BEP 是双向的——两个设备既是 seeder 又是 leecher。"

### Q3: 如何保证与 Go 版本的线级兼容？（⭐⭐⭐⭐）

**推荐回答**：
"关键点：
1. **Protobuf 字段编号和类型完全一致**：wire format 的字节布局都必须一致
2. **TLS 握手行为一致**：Certificate 格式、设备 ID 派生方式（SHA-256 of public key → base32）
3. **Block Size 默认值一致**：128KB，因为块哈希是整个协议的基础
4. **压缩算法**：必须用相同的 LZ4 变体

验证：跨版本集成测试——Go Syncthing + Rust 实例配置同一文件夹，验证双向同步。

遇到的主要坑：Protobuf 的字段默认值处理。Go protobuf 库和 Rust prost 对默认值的序列化行为不同，有时 Go 版本省略了默认值字段而 prost 期望显式出现。"

### Q4: 为什么选 Ed25519 而非 RSA？（⭐⭐⭐）

**推荐回答**：
"1. 密钥更短：Ed25519 私钥 32 bytes vs RSA 2048-bit 私钥 ~256 bytes
2. 签名更快：Ed25519 签名速度远超 RSA，在 P2P 频繁握手场景中收益明显
3. 安全级别相当：Ed25519 提供 ~128-bit 安全级别
4. 无 timing attack 风险：Ed25519 是确定性的，不存在 RSA 的 padding oracle 类攻击
5. Rust 生态：ed25519-dalek 是纯 Rust 的高质量实现，Ring 也原生支持

设备 ID 派生：DeviceID = base32(SHA-256(Ed25519PublicKey))，32 字符的短标识符。"

### Q5: sled vs SQLite 的选型？（⭐⭐⭐⭐）

**推荐回答**：
"选 sled 而非 SQLite 的原因：
1. **纯 Rust**：零 C 依赖，保持'零运行时依赖'承诺
2. **COW B+Tree**：读多写少的元数据场景下性能极佳。写操作复制修改节点，不影响并发读
3. **API 友好**：`db.insert(key, value)` 相比 SQL 更直接
4. **无需 schema 迁移**：KV 存储天然 schema-less

但 sled 已停止维护，未来可能迁移到 redb 或 libsql。架构上通过 `BlockStore` trait 隔离具体实现，迁移不影响上层。"

### Q6: 四种发现机制的协同？（⭐⭐⭐⭐）

**推荐回答**：
"按优先级：
1. **LAN UDP 广播（最快）**：局域网内发 UDP broadcast 255.255.255.255:21027，其他设备响应。无需外部服务
2. **STUN（中等）**：获取公网 IP:Port，帮助 NAT 后的设备建立直连
3. **UPnP/NAT-PMP（中等）**：在路由器上自动创建端口映射
4. **Global Discovery（最慢但全局可达）**：向 discovery.syncthing.net 注册和查询。任何地方都能找到设备
5. **Relay（兜底）**：如果以上都失败，通过中继服务器转发流量

实现上，ParallelDialer 同时尝试所有发现的地址，第一个成功的连接获胜。"

### Q7: 块级同步的完整流程？（⭐⭐⭐⭐⭐）

**推荐回答**：
"1. Scanner 遍历本地文件夹，每个文件按 128KB 分块，用 rayon 并行计算 SHA-256
2. 生成 Index 消息发给所有连接的设备
3. 收到对端 Index 后，FolderModel 对比本地和远程的块哈希列表
4. 找出缺失的块，发送 Request 消息（批量）
5. 接收 Response，验证 SHA-256，写入临时文件
6. 所有块接收完毕后，原子 rename 到目标路径
7. 如果文件同时被双方修改，触发三路合并（similar crate）

关键优化：concurrency_policy 根据 RTT 动态调整并发请求数——低延迟高并发，高延迟低并发。避免拥塞。"

### Q8: 零运行时依赖的代价？（⭐⭐⭐）

**推荐回答**：
"好处：
- 单静态二进制 ~13MB，`scp` 到另一台机器就能跑
- 无 DLL hell、容器依赖
- 安全审计面小

代价：
- 不能利用系统 CA 证书（用 rustls-native-certs 桥接）
- 需要自己维护 TLS/加密/lz4 等依赖的安全更新
- 编译时间长（LTO + single codegen unit）

总体来说收益大于代价——特别是对于需要跨平台部署的 P2P 工具。"

### Q9: 三路合并冲突解决？（⭐⭐⭐）

**推荐回答**：
"场景：文件同时被本地和远程修改。

三路合并：
1. **Base**：文件在双方最后一次同步时的共同版本（通过版本向量找到）
2. **Local**：本地当前版本
3. **Remote**：远程当前版本

用 similar crate 执行文本级三路合并，找出共同的、只在一方改变的、双方都改变的行。

合并成功 → 写入合并结果。合并失败 → 保留两个冲突副本：`filename.sync-conflict-2026-06-27-120000.ext`。

对于二进制文件，三路合并不适用——直接保留较新的版本。"

## 3. 与 Go Syncthing 的对比

> "对比维度：二进制大小（13MB vs ~10MB）、内存占用（低，无 GC 停顿）、启动时间、代码复杂度。Rust 版虽然代码量更多（静态类型标注），但运行时行为更可预测——没有 GC 导致的延迟尖峰，没有 goroutine 泄漏风险。在长时间运行场景下，这一点很重要。"

## 4. 技术深挖问题

### 深挖1：现场设计一个简单的 P2P 发现协议

**提示**："假设只有 UDP、没有中心服务器，如何设计一个设备发现协议？"

**推荐回答要点**：广播+响应模式、TTL 限制、随机退避避免广播风暴、缓存已发现设备减少重复响应、考虑安全（如何防止伪造响应）。

### 深挖2：如何优化大文件夹的首次同步？

**推荐回答要点**：分批索引（避免内存峰值）、优先级调度（小文件优先，感知上更快）、压缩传输、并行请求多个块、增量扫描后跳过未变化文件。
