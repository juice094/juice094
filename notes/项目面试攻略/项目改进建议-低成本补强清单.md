# 项目改进建议：低成本补强清单

> **用途**：记录 Clarity / devbase / syncthing-rust 三项目可优先落地的技术补强方向，既提升项目真实技术深度，也增强简历可防御性。
> **原则**：不要为了简历而做技术；优先做**低成本、高价值、能讲清楚**的事。

---

## 一、总体判断

当前三个项目的技术深度**已经足够支撑有竞争力的简历**。后续改进的重点不是追逐更多高级术语，而是：

1. **把已实现的技术显式化、文档化、可验证**
2. **补齐少量低成本的工程抽象和 benchmark**
3. **修正现有代码与简历描述之间的不一致**

---

## 二、Clarity 项目改进建议

### 2.1 Agent Loop 状态机显式化（高价值 / 低成本）

**现状**：`clarity-core/src/agent/` 已有 `plan.rs`、`driver.rs`、`executor.rs`、`run.rs` 等文件，状态转换逻辑可能已隐含在代码中。

**建议**：
- 将 ReAct / Plan 的状态流转显式建模为一个 `AgentState` 枚举 + `transition` 方法
- 绘制状态图并写入 `docs/agent-state-machine.md`
- 为每个状态转换添加单元测试

**收益**：
- 代码可维护性提升
- 面试时可清晰讲解"ReAct 和 Plan 如何切换、失败如何回退"
- 简历可写"基于显式状态机的双模式 Agent loop"

**预计投入**：1-2 天

---

### 2.2 Plan 模式步骤遗漏率评估方法（高价值 / 中成本）

**现状**：简历写"Plan 模式显著降低长任务步骤遗漏率"，但缺少评估方法论。

**建议**：
- 定义一个 benchmark 数据集：20-50 个多步骤任务（如"重构模块 A 并更新测试"）
- 用 ReAct 模式和 Plan 模式分别执行，记录完成任务所需的 LLM 调用次数和步骤完成率
- 输出一份 `docs/agent-evaluation.md`

**收益**：
- 把"显著降低"从主观感受变成可复现数据
- 面试被问及时能给出具体测试方法

**预计投入**：3-5 天

---

### 2.3 记忆同步机制边界清晰化（高价值 / 低成本）

**现状**：`clarity-memory` 只做本地存储，真正的跨设备同步在 `clarity-claw` 的 mesh 层，且只在 RoleContext 级别。

**建议**：
- 在 `projects/clarity.md` 和 `AGENTS.md` 中明确：
  - `clarity-memory` = 本地记忆存储与检索
  - `clarity-claw` mesh = 跨设备角色上下文同步
  - memory-level 联邦同步 = 后续方向
- 删除"记忆联邦与脱敏合并"等未实现的表述

**收益**：
- 消除简历最大风险点
- 明确架构边界，便于后续扩展

**预计投入**：0.5-1 天（文档修正）

---

### 2.4 端侧推理量化 benchmark（中价值 / 中成本）

**现状**：`clarity-llm` 支持 Candle/GGUF/Ollama，但没有量化对比数据。

**建议**：
- 选取 2-3 个典型模型（如 Qwen2.5-3B、Llama-3.2-3B）
- 测试 FP16、Q8_0、Q4_K_M 三种格式在目标设备上的：
  - 加载内存占用
  - 推理 tokens/sec
  - 简单任务准确率 / perplexity
- 输出 `docs/local-llm-benchmark.md`

**收益**：
- 简历可写真实量化数据（如"Q4_K_M 降低 X% 内存，推理速度提升 Y%"）
- 避免编造 60% 这类数字

**预计投入**：3-5 天

---

### 2.5 删除或迁移冗余 candle 依赖（低价值 / 低成本）

**现状**：`clarity-llm/Cargo.toml` 声明了 candle-core/candle-transformers，但实际代码可能通过 kalosm 间接使用。

**建议**：
- 确认 candle 是否被直接使用
- 如果没有，从 dependencies 中移除或改为 optional

**收益**：
- 减少编译时间和依赖面
- 避免简历上写"Candle"但实际上没用

**预计投入**：0.5 天

## 三、devbase 项目改进建议

### 3.1 tree-sitter SymbolExtractor 统一抽象（高价值 / 低成本）

**现状**：`src/semantic_index/symbol.rs` 已支持多语言解析，但可能没有统一的 trait 门面。

**建议**：
- 定义一个 `SymbolExtractor` trait：
  ```rust
  pub trait SymbolExtractor {
      fn extract_symbols(&self, source: &str, path: &Path) -> Vec<CodeSymbol>;
      fn supported_languages(&self) -> &[Language];
  }
  ```
- 为 Rust/Python/TypeScript/Go 分别实现
- 写一个 `docs/symbol-extraction.md` 说明设计

**收益**：
- 代码更模块化
- 简历可写"基于统一 SymbolExtractor 抽象的多语言代码解析"
- 面试可讲 trait 设计与语言差异屏蔽

**预计投入**：1-2 天

---

### 3.2 MCP 工具文档化与分类（高价值 / 低成本）

**现状**：实际有 81 个 MCP 工具，但简历只写数量，缺少分类和代表性工具说明。

**建议**：
- 将 81 个工具按功能分类：代码检索、笔记管理、Skill 执行、工作流、外部 API
- 挑选 5-10 个最具代表性的工具，写出其输入/输出/使用场景
- 生成 `docs/mcp-tools.md`

**收益**：
- 面试被问"81 个工具都做什么"时有话说
- 简历可写"覆盖 X/Y/Z 五类场景的 81 个 MCP 工具"

**预计投入**：1-2 天

---

### 3.3 混合检索评估 benchmark（高价值 / 中成本）

**现状**：BM25 + vector 混合检索已实现，但没有评估数据。

**建议**：
- 准备一个评估数据集：50-100 个代码/笔记查询，标注期望结果
- 分别测试：纯 BM25、纯 vector、简单加权融合
- 记录 Hit Rate@K 和 MRR
- 输出 `docs/retrieval-evaluation.md`

**收益**：
- 有真实数据支撑"混合检索优于单路检索"
- 面试可讲评估方法论

**预计投入**：3-5 天

---

### 3.4 Workspace crate 边界审查（中价值 / 低成本）

**现状**：12 个 workspace crates 基本合理，但个别 crate（如 `devbase-symbol-links`）职责可能不够清晰。

**建议**：
- 为每个 crate 写一句职责描述
- 检查是否存在循环依赖
- 输出 `docs/crate-ownership.md`

**收益**：
- 简历写"12 crate Workspace 拆分"时更有底气
- 便于后续贡献者理解架构

**预计投入**：1 天

## 四、syncthing-rust 项目改进建议

### 4.1 证书算法升级或表述修正（中价值 / 低成本）

**现状**：源码使用 rcgen 默认 ECDSA P-256 生成自签名证书，但 `Cargo.toml` 仍依赖未使用的 `ed25519-dalek`。

**建议**：
- 方案 A（推荐）：保持 ECDSA P-256，从 `Cargo.toml` 移除 `ed25519-dalek` 依赖，简历写"rustls + 自签名设备证书"
- 方案 B：若希望简历写 ed25519，则改造证书生成逻辑使用 ed25519-dalek

**收益**：
- 消除依赖冗余
- 简历表述与代码一致

**预计投入**：0.5 天（方案 A）或 2-3 天（方案 B）

---

### 4.2 三路合并与版本归档评估（中价值 / 中成本）

**现状**：三路合并和 Staggered 版本归档已实现，但没有对比数据。

**建议**：
- 构造 50-100 组并发修改测试用例
- 对比：无版本归档、Simple 归档、Staggered 归档下的数据丢失率 / 冲突解决率
- 输出 `docs/conflict-resolution-evaluation.md`

**收益**：
- 简历可写"Staggered 归档在 X 场景下降低数据丢失风险"
- 避免空泛的"降低误冲突回滚率"

**预计投入**：3-5 天

---

### 4.3 NAT 穿透成功率测试（中价值 / 中成本）

**现状**：STUN/UPnP/Relay 均已实现，但没有在各种 NAT 类型下的成功率数据。

**建议**：
- 在多种网络环境下测试（公网、锥形 NAT、对称 NAT、企业防火墙）
- 记录直连成功率、UPnP 成功率、Relay fallback 比例
- 输出 `docs/nat-traversal-evaluation.md`

**收益**：
- 面试可讲"为什么需要三种穿透机制"
- 简历可写"覆盖 X 种网络拓扑的 NAT 穿透方案"

**预计投入**：2-3 天

---

### 4.4 完成 72h 耐久压测（高价值 / 高成本）

**现状**：`scripts/churn_72h.ps1` 脚本存在，但尚未完成。

**建议**：
- 完成压测脚本并跑通 72 小时
- 记录内存、CPU、同步延迟、错误率等指标
- 输出 `docs/72h-stress-test.md`

**收益**：
- 简历可真实写"通过 72h 耐久压测"
- 项目质量接近生产级

**预计投入**：1-2 周

## 六、不建议做的事

| 事项 | 原因 |
|:---|:---|
| 为写 RRF 而去实现 RRF | 当前 BM25 + vector 已够用，边际收益低 |
| 把 SQLite 换成 MVCC 数据库 | 与本地优先、零依赖定位冲突 |
| 把 devbase 拆成 22 个 crates | crate 数量不是越多越好，协调成本上升 |
| 编造具体数字（60%、30%、降低 X%） | 面试一问就穿帮，损害可信度 |
| 在 Clarity 中做 memory-level CRDT | 工作量大，且当前 RoleContext 同步已能满足核心场景 |
