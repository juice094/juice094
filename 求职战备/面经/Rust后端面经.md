## 3. 技术面高频考点

### 第一轮：Rust 语言深度（40 min）

**必问题**：
1. 解释所有权和借用的核心规则，举例说明 NLL 解决的问题
2. Send 和 Sync 的含义，为什么 Rc 不是 Send、RefCell 不是 Sync
3. static dispatch vs dynamic dispatch，什么时候用 Box<dyn Trait>
4. async/await 的原理，Future trait 和 Pin 的关系

**推荐回答策略**：直接给出结论 → 递进深入细节 → 用项目中的实际案例佐证。

示例："为什么项目中用 parking_lot 而非 std::sync？—— 三个原因：1) 性能更好（用户态自旋减少系统调用）；2) 无中毒机制更符合实际使用（panic不应传播到其他线程的锁等待）；3) 更丰富的API。在 Clarity 的 clarity-core 中，几乎所有 Mutex/RwLock 都用 parking_lot。"

### 第二轮：项目深挖（45 min）

**面试官会如何追问 Clarity**：
- "22 个 crate 的依赖关系怎么管理？Circular dependency 怎么避免？" → Contract-First 架构，clarity-contract 零依赖在最底层
- "为什么选择自己实现而不是用 LangChain？" → 本地优先、Rust 生态、单二进制部署
- "SPMC 为什么不用现成的消息队列？" → 嵌入式场景不需要独立 MQ 服务，SPMC 广播语义更匹配

**面试官会如何追问 Syncthing-rust**：
- "为什么重新实现而不是贡献 Go 版本？" → 学习动机 + 工程需求（嵌入 Rust 项目）+ 能力证明
- "与 Go 版本线级兼容的最大挑战？" → Protobuf 字段默认值处理差异
- "sled 的事务模型？为什么选它？" → COW B+Tree，读多写少最优，但已停止维护，未来迁移计划

### 第三轮：系统设计（45 min）

针对候选人背景，面试官可能出的系统设计题：
- "设计一个类似 Clarity 的本地 Agent 运行时" → 你已经做了，关键是如何清晰表达
- "设计一个 MCP 工具网关" → 服务发现 + 路由 + 审批 + 审计
- "设计一个代码搜索引擎" → tree-sitter + Tantivy + 向量 + RRF 融合

### 第四轮：主管/交叉面（30 min）

主管更关注：技术判断力、架构品味、学习能力。

## 6. 心理建设

### "你是本科生，凭什么和研究生竞争？"

> "我的三个项目——22 crate 的 Clarity、71 MCP 工具的 Devbase、BEP 协议栈的 Syncthing-rust——每个都达到生产级质量标准（零 Clippy 警告、1554+ 测试、CI/CD 全覆盖）。研究生培养的是特定方向的深度，而我培养了从零到一交付系统的广度。在很多公司，能独立把想法变成可运行系统的人比有论文但缺乏工程能力的人更有价值。"

### "你的学校不是 985/211"

> "我的 GitHub 是我真正的简历。开源项目的质量不受学校影响——代码不会说谎。1554 个通过的测试比学校排名更能证明我的能力。"

### "你没有实习经历，怎么适应团队开发？"

> "我独立管理 3 个大型项目的完整生命周期——从架构设计到 CI/CD 到文档。虽然缺乏正式的 code review 流程和生产环境运维经验，但我通过强制执行 clippy/fmt/audit 质量门禁、编写详细的 ARCHITECTURE 文档来模拟团队开发的最佳实践。在 Clarity 中，我甚至编写了架构守卫规则（类似生产团队的 lint rule）来防止架构退化。加入团队后，我最大的期待就是把这些个人实践升级为团队级的工程规范。"

### 面试前的心理准备

1. **准备决定心态**：最充分的准备 = 最自然的自信
2. **面试是双向选择**：你也在评估团队是否适合你
3. **被拒不是否定你这个人**：可能只是不匹配
4. **每一轮面试都是免费的学习机会**：面试官的追问是技术成长的最好养料
