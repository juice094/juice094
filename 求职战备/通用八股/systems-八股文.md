## 1. 系统设计方法论

### 1.1 需求澄清框架

**功能性需求**：系统做什么（注册、搜索、同步、推荐...）
**非功能性需求**：
- 可用性（99.9%? 99.99%?）
- 延迟（P50/P99 < X ms?）
- 吞吐量（QPS 目标？）
- 一致性（强一致/最终一致？）
- 安全（认证/授权/加密？）

### 1.2 容量估算

```
QPS 估算：日活用户 × 人均请求数 / 86400
存储估算：用户数 × 人均数据量 × 副本数 × 预留增长
带宽估算：QPS × 平均响应大小
内存估算：热点数据 × 1.2（元数据开销）
```

### 1.3 关键概念速查

| 概念 | 一句话 | 适用场景 |
|------|--------|---------|
| **垂直扩展** | 更强的机器 | 单体、低 QPS |
| **水平扩展** | 更多的机器 | 高 QPS、无状态 |
| **数据库分片** | 按 key 分布数据 | 单库扛不住 |
| **主从复制** | 写主读从 | 读多写少 |
| **CDN** | 就近缓存 | 静态资源、大文件 |
| **消息队列** | 异步解耦 | 耗时任务、削峰 |
| **缓存** | 热点数据快速访问 | 读多、变化少 |

## 2. 高频系统设计题

### 2.1 设计一个 AI Agent 平台（类似 Coze/Dify）

**需求分析**：
- 功能：Agent 创建配置、工具集成、知识库、对话管理、工作流编排
- 非功能：低延迟流式响应（< 2s 首字）、多租户隔离、可用性 99.9%

**整体架构**：
```
┌─────────────────────────────────────────────┐
│                  API Gateway                 │
│         (认证/限流/路由/日志)                │
├─────────────────────────────────────────────┤
│   Agent Service  │  Knowledge Service       │
│   (Agent引擎+    │  (向量DB+                │
│    工具调用+     │   文档解析+              │
│    审批链)      │   混合搜索)              │
├─────────────────────────────────────────────┤
│   Workflow Engine│  Model Router            │
│   (DAG执行器)   │  (多Provider+            │
│                  │   负载均衡+熔断)         │
├─────────────────────────────────────────────┤
│   Conversation  │  Tool Registry           │
│   Store (SQL)   │  (MCP/内置工具)          │
└─────────────────────────────────────────────┘
```

**关键设计决策**：
- Agent 引擎：ReAct 循环 + Plan 模式双模
- 流式响应：SSE/WebSocket，首 token 到达立即推送
- 工具安全：分层审批（Interact/Smart/Yolo）+ 沙盒隔离
- 记忆系统：三层（短期上下文 + 工作记忆 + 长期向量检索）
- 多租户：每个租户独立的 Agent 配置和知识库

### 2.2 设计一个代码语义搜索引擎

**需求**：自然语言搜索代码、符号跳转、相似代码推荐

**索引 Pipeline**：
```
代码仓库 → tree-sitter 解析 → 符号提取 → 
├─ 全文索引 (Tantivy/ES BM25) → 关键词搜索
└─ 向量嵌入 (Embedding Model) → 语义搜索
→ RRF 融合 → 搜索结果
```

**增量更新**：git hook → 只重索引变化的文件

**搜索流程**：
1. 用户输入自然语言查询
2. 查询 embedding → 向量检索 Top 20
3. 查询 → BM25 分词 → 全文检索 Top 20
4. RRF 融合 → 返回 Top 10
5. 高亮匹配行 + 文件预览

### 2.3 设计一个 P2P 文件同步系统

**需求**：多设备自动同步、冲突解决、离线支持、端到端加密

**设备发现**：LAN UDP 广播 + STUN 公网发现 + Global Discovery Server + Relay 中继兜底

**同步流程**：
1. 文件变更触发 Scanner（SHA-256 分块哈希）
2. 生成 Index → 发送给所有已连接设备
3. 对端对比 Index → 找出缺失块 → Request
4. Response 传输块数据 + LZ4 压缩
5. SHA-256 验证 → 原子 rename
6. 冲突检测 → 三路合并

**关键设计**：
- 块级同步：比文件级更高效（修改大文件的 1% 只需传 1%）
- 自适应并发：根据 RTT 调整并行块请求数
- 版本向量：追踪文件历史，用于冲突解决

### 2.4 设计一个 MCP 工具网关

**需求**：统一管理多个 MCP 服务、负载均衡、权限控制、安全审计

**架构**：
```
AI Agent → MCP Gateway (认证/路由/审计)
                ├→ MCP Server A (文件工具)
                ├→ MCP Server B (数据库工具)
                └→ MCP Server C (云端工具)
```

**关键功能**：
- 服务发现和健康检查
- 工具聚合（合并多个 server 的 tool list）
- 请求路由（按 tool name 分发到对应 server）
- 权限控制（Agent 身份 → 可用工具白名单）
- 审批集成（高风险工具需要额外审批）
- 审计日志（所有工具调用记录）

## 3. 数据库设计专题

### 3.1 索引原理

**B+Tree**：
- 所有数据在叶子节点，内部节点只存 key 用于导航
- 叶子节点通过链表连接 → 范围查询高效
- 查询复杂度 O(log_B N)，B 是分支因子

**覆盖索引**：查询的所有列都在索引中 → 不需要回表
**联合索引最左前缀**：`INDEX(a, b, c)` → 能加速 `WHERE a=?` / `a=? AND b=?`，不能加速 `WHERE b=?`

### 3.2 事务隔离级别

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|----------|------|
| Read Uncommitted | 是 | 是 | 是 |
| Read Committed | 否 | 是 | 是 |
| Repeatable Read | 否 | 否 | 是(MySQL: 否) |
| Serializable | 否 | 否 | 否 |

**脏读**：读到未提交的事务的修改
**不可重复读**：同一事务内两次读取结果不同（被其他事务 update 提交）
**幻读**：同一事务内两次范围查询结果不同（被其他事务 insert 提交）

### 3.3 MVCC 原理

Multi-Version Concurrency Control：每行数据有多个版本。读操作看到事务开始时的快照，写操作创建新版本。读写互不阻塞。

PostgreSQL 用事务 ID (XID) 判断可见性，MySQL InnoDB 用 undo log + Read View。

## 4. 分布式系统专题

### 4.1 CAP 理论

一个分布式系统最多同时满足三项中的两项：
- **C (Consistency)**：所有节点同一时刻数据一致
- **A (Availability)**：每个请求都收到响应（不保证数据最新）
- **P (Partition Tolerance)**：网络分区时系统继续工作

实际选择：P 必须选（网络分区不可避免），在 C 和 A 之间权衡。

### 4.2 一致性模型

| 模型 | 含义 | 典型系统 |
|------|------|---------|
| 强一致 | 写入后所有读返回最新值 | Paxos/Raft, Zookeeper |
| 最终一致 | 一段时间后所有副本一致 | DNS, Cassandra |
| 因果一致 | 有因果关系的操作按序可见 | DynamoDB, CRDT |

### 4.3 分布式锁

**Redis 实现**：`SET lock_key unique_id NX EX 30`（原子设置 + 过期时间）
释放时检查 unique_id，避免误删别人的锁。

**Redlock**：在 N 个独立 Redis 实例上获取锁，超过半数成功才认为获取成功。争议：Martin Kleppmann 认为不够安全。

## 5. 性能优化专题

### 5.1 缓存策略

| 策略 | 读 | 写 |
|------|----|-----|
| Cache-Aside | 读 cache → miss → 读 DB → 写 cache | 写 DB → 失效 cache |
| Read-Through | 读 cache → miss → cache 自动读 DB | 不适用 |
| Write-Through | — | 写 cache → cache 同步写 DB |
| Write-Behind | — | 写 cache → 异步写 DB |

### 5.2 缓存三大问题

- **穿透**：查不存在的数据，每次都打到 DB → 布隆过滤器 / 缓存空值
- **击穿**：热点 key 过期，大量请求打到 DB → 互斥锁 / 永不过期 + 异步更新
- **雪崩**：大量 key 同时过期 → 随机 TTL / 多级缓存 / 限流降级

## 6. 安全专题

### API 认证与授权

- **JWT**：无状态，包含 claims + 签名。适合微服务间认证。缺点：token 签发后无法主动失效
- **OAuth2**：授权码模式 → 客户端获得 access token → 访问资源服务器。第三方登录首选
- **mTLS**：双向 TLS 认证。双方各出示证书。适合服务间通信（候选人 syncthing-rust 中使用）

### 常见攻击

| 攻击 | 防御 |
|------|------|
| SQL Injection | 参数化查询，ORM |
| XSS | 输出编码，CSP |
| CSRF | CSRF Token，SameSite Cookie |
| SSRF | URL 白名单，禁用内网 IP |
