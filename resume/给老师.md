# 给辅导老师的说明

## 这是什么

学生周景潇的简历草稿包。包含三份材料：

| 文件 | 用途 |
|:---|:---|
| `zh.md` | **主力中文简历**——这是最终需要您把关的核心文档 |
| `en.md` | 英文版简历，内容对应中文版 |
| `resume-assets-combined.md` | **项目技术资产全量清单**——三个项目的模块拆解、架构模式、量化数据、解决过的硬核技术问题、可直接用的 bullet points |

## 学生的困境

学生长期独立开发（vibe coding），对自己的项目做到了什么程度、用了什么技术栈、有哪些值得写在简历上的点——心里没数。这次趁着完善简历，第一次系统梳理了三个 Rust 项目的全貌。

**所以这份草稿存在一个问题**：学生不知道从业者想看什么，您不知道学生具体做了什么。这份资产清单就是用来填这个信息差的——它包含了三个项目能说的所有技术细节，请您用行业经验判断哪些值得写、哪些该砍、哪些表述不专业。

## 需要您重点把关的

1. **项目描述的行业术语**：学生用的是开源社区语言，不一定符合企业招聘的表述习惯。比如"SPMC 事件总线"在企业 JD 里可能叫"消息驱动架构"——这类的翻译需要您把关

2. **量化数据的可信度**：简历里写了测试数量、代码行数、性能指标。这些是真实数据，但表述方式是否合理（比如写"1,889 测试全通过"好还是"全量测试通过"好）——需要您判断

3. **研究经历的包装**：论文还没发表，具体发现不能公开写。目前用方法论语言写的——强调实验设计、统计管道、观测框架——您看这个抽象层次是否合适

4. **技能区的取舍**：学生主力 Rust，Python 用于研究。TypeScript/Vue/Go 只是课程接触——已经删掉了。是否还需要补充其他技能？

5. **整体结构**：目前的顺序是 简介→教育→技能→研究→项目→认证。对一个大三学生来说，研究放在项目前面是否合适？

## 关于项目

三个项目均为独立开发，GitHub 公开仓库：
- **Clarity**（22 crates, 1,889 tests）：本地 AI Agent 运行时
- **syncthing-rust**（13 crates, 392 tests）：P2P 文件同步引擎
- **devbase**（12 crates, 71 MCP tools）：开发者工作空间编译器

如有需要，可以直接查看仓库代码。简历里的每一个数据点都在资产清单里有来源标注。
